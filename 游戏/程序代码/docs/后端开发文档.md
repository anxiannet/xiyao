## 4. API接口清单

所有接口统一返回：

```ts
type ApiResponse<T> = {
  success: boolean;
  data: T | null;
  error: ApiError | null;
  debug?: unknown;
};
```

`debug` 只允许开发模式返回。

### 4.1 POST /matches

创建对局。

请求：

```ts
type CreateMatchRequest = {
  mode: 'player_vs_ai' | 'ai_vs_ai';
  map_id: string;
  first_side: 'qingqiu' | 'tianmen';
  qingqiu_squad_id: string;
  tianmen_squad_id: string;
};
```

流程：

1. 读取 `xiyao_cfg_maps`。
2. 读取 `xiyao_cfg_map_tiles`。
3. 读取双方 squad、unit、skill 配置。
4. 生成 `xiyao_match_matches`。
5. 生成 `xiyao_match_tiles` 快照。
6. 初始化 `xiyao_match_units`。
7. 写入 `match_start`、`round_start`、`turn_start` 事件。
8. 返回完整对局状态。

### 4.2 GET /matches/:id

读取完整对局状态。

返回内容：

- match
- units
- tiles
- statuses
- current_turn
- score
- winner

### 4.3 POST /matches/:id/actions

提交玩家动作。

请求字段必须包含：

```ts
type MatchActionRequest = {
  match_id: string;
  unit_id: string | null;
  action_type: 'move' | 'attack' | 'skill' | 'capture' | 'end_turn';
  source_tile_id: string | null;
  target_tile_id: string | null;
  target_unit_id: string | null;
  skill_id: string | null;
  payload: Record<string, unknown> | null;
};
```

`payload` 只能放调试辅助信息，不允许承载规则逻辑。

### 4.4 POST /matches/:id/ai-action

为当前行动方执行一次 AI 动作。

流程：

1. 读取 RuleState。
2. `ActionGenerator` 生成合法候选动作。
3. `AIScorer` 根据 `AI行为评分表.md` 评分。
4. 选择最高分动作。
5. 写入 `ai_decision` 事件。
6. 通过 `RuleEngine.executeAction` 执行动作。
7. 返回最新对局状态。

若 `AI行为评分表.md` 未建立，接口可以返回 `AI_SCORER_NOT_CONFIGURED`，或使用测试桩评分，但测试桩不得进入稳定平衡结论。

### 4.5 POST /matches/:id/auto-run

自动跑完整局。

请求：

```ts
type AutoRunRequest = {
  max_steps: number;
};
```

约束：

- 每步只调用一次 `ai-action` 流程。
- 不允许跳过规则引擎直接改结果。
- 达到胜负、非法状态或 `max_steps` 时停止。

### 4.6 POST /simulations/ai-vs-ai

批量 AI vs AI 测试。

请求：

```ts
type AIVsAIRequest = {
  map_ids: string[];
  total_matches: number;
  first_side_policy: 'alternate' | 'qingqiu_first' | 'tianmen_first';
  max_steps_per_match: number;
};
```

必须支持 1000 局统计。

统计字段：

- winner_side
- round_count
- kill_count
- capture_count
- skill_count
- status_trigger_count
- decree_count
- map_id
- first_side

### 4.7 GET /matches/:id/events

读取事件日志。

支持参数：

```text
?from_sequence=1&limit=200
```

## 5. 规则引擎流程

每次动作必须走统一入口：

```ts
RuleEngine.executeAction(ruleState, action): RuleEngineResult
```

执行顺序：

1. 校验对局状态。
2. 校验行动方。
3. 校验单位是否存在。
4. 校验单位是否可行动。
5. 校验动作是否合法。
6. 执行动作。
7. 结算移动、伤害、技能、状态、据点、回合推进。
8. 写入事件。
9. 返回最新对局状态。

禁止：

- API 直接修改对局结果。
- 前端传入伤害、命中、胜负结果。
- AI 直接改状态。
- 数据库字段内嵌规则判断。

## 6. 合法动作范围

当前只支持：

```text
move
attack
skill
capture
end_turn
```

不得新增动作类型。

### 6.1 move

校验：

- 当前单位属于行动方。
- 单位存活。
- 单位仍可行动。
- `source_tile_id` 为单位当前格。
- `target_tile_id` 可进入。
- 路径不穿过障碍或被占格阻挡。
- 移动距离符合当前 MV 与状态修正。

输出事件：

- `move`
- 可能产生 `status_add`、`status_remove`

### 6.2 attack

校验：

- 攻击者存活且可行动。
- 目标单位存在且可被攻击。
- 距离与视线合法。
- 支援来源由引擎计算。

输出事件：

- `attack`
- `damage`
- 可能产生 `status_add`、`status_remove`
- 若击败单位，更新气运与单位状态

### 6.3 skill

校验：

- `skill_id` 属于该单位。
- AP 足够。
- 目标合法。
- 技能效果由规则引擎读取配置和规则本体执行。

输出事件：

- `skill`
- 后续按实际结算产生 `damage`、`status_add`、`capture` 等事件

### 6.4 capture

校验：

- 单位位于据点范围。
- 只统计正式英雄单位。
- 执行方正式单位数量大于敌方正式单位数量。

输出事件：

- `capture`

### 6.5 end_turn

校验：

- 当前行动方合法。
- 当前单位或当前方可以结束行动。

输出事件：

- `turn_end`
- 必要时 `round_end`
- 必要时 `round_start`
- 必要时 `turn_start`
- 必要时 `match_end`

错误码必须区分：

```text

ILLEGAL_UNIT
ILLEGAL_TURN
RULE_RESOLUTION_FAILED
DATABASE_ERROR

```

错误响应：

```ts
type ApiError = {
  code: string;
  message: string;
  details?: unknown;
};
```

## 8. AI后端接入流程

AI 不得调用 ChatGPT、外部 AI 模型、联网服务。

AI 只能调用：

```text
ActionGenerator
AIScorer
RuleState
```

### 8.1 ActionGenerator

输入：

```ts
type ActionGeneratorInput = {
  rule_state: RuleState;
  side: 'qingqiu' | 'tianmen';
};
```

输出：

```ts
type CandidateAction = {
  action: MatchActionRequest;
  legal: boolean;
  reason: string | null;
};
```

只允许生成合法动作候选；非法候选只可用于调试，不可执行。

### 8.2 AIScorer

输入：

```ts
type AIScorerInput = {
  rule_state: RuleState;
  candidates: CandidateAction[];
};
```

输出：

```ts
type ScoredAction = {
  action: MatchActionRequest;
  total_score: number;
  score_breakdown: Array<{
    key: string;
    value: number;
    reason: string;
  }>;
};
```

评分依据必须来自 `游戏/程序代码/docs/AI行为评分表.md`。

### 8.3 AIDecisionService

输出必须包含：

- 候选动作
- 每个动作评分
- 最终选择
- 评分组成

并写入 `ai_decision` 事件。

## 9. AI vs AI模拟流程

单局流程：

1. `POST /matches` 创建 `ai_vs_ai` 对局。
2. 循环调用 `AIDecisionService`。
3. 每次决策都通过 `RuleEngine.executeAction`。
4. 每步写入事件。
5. 对局结束后写入 `xiyao_match_ai_stats`。
6. 返回单局统计。

批量流程：

1. 读取地图列表。
2. 按 `first_side_policy` 分配先手方。
3. 重复创建并运行对局。
4. 每局写入统计。
5. 汇总胜率、平均回合、击杀、占点、技能、状态、律令次数。

1000 局测试必须可复现：

- 记录 `random_seed`。
- 记录 `engine_version`。
- 记录 `ai_version`。
- 记录地图 ID 与先手方。

## 10. 测试用例清单

### 创建对局测试

- 可创建 `player_vs_ai`。
- 可创建 `ai_vs_ai`。
- 地图不存在时返回 `MATCH_NOT_FOUND` 或配置错误。
- 单位初始化数量与配置一致。
- 地图快照数量与配置一致。
- 初始事件包含 `match_start`、`round_start`、`turn_start`。

### 移动测试

- 合法移动成功。
- 障碍不可进入。
- 超出移动距离失败。
- 非当前行动方失败。
- 迷踪触发后移动距离降低且移除。

### 攻击测试

- 合法攻击成功。
- 目标不存在失败。
- 距离不合法失败。
- 支援由引擎计算。
- 击败单位后生成伤害与胜利分相关更新。

### 技能测试

- 单位只能使用自己的核心技能。
- AP 不足失败。
- 技能目标非法失败。
- 技能事件与后续状态事件完整生成。

### 占领测试

- 据点范围内正式单位优势时占领成功。
- 数量相同占领失败。
- 假身不计入占领人数。
- 据点归属保持直到被明确占领。

### 结束回合测试

- 合法结束行动。
- 回合结束触发状态结算。
- 大回合结束触发据点气运结算。
- 第 3 大回合后触发胜负结算。

### AI动作测试

- AI 只能从合法候选动作选择。
- AI 决策写入 `ai_decision`。
- AI 输出候选动作、评分、最终选择、评分组成。
- `AI行为评分表.md` 缺失时不产生稳定评分结论。

### AI vs AI完整局测试

- 单局可自动结束。
- `max_steps` 可阻断异常死循环。
- 每局写入 `xiyao_match_ai_stats`。
- 批量 1000 局可生成统计。
