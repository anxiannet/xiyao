# 《夕妖》MVP数据结构字段表

## 文档用途

本文档属于开发文档。

它用于把 `游戏/游戏设定/` 中的 MVP 规则转换成程序可实现的数据结构。

当前阶段只定义：

- 表名前缀
- 表类型
- 表用途
- 表之间的关系
- 是否进入当前 MVP

字段细节后续在本文档继续展开。

---

# 1. 文档边界

## 设定文档与开发文档分工

| 目录 | 用途 | 示例 |
|---|---|---|
| `游戏/游戏设定/` | 设定文档，定义游戏规则、世界观、小队、自由对战规则 | 规则总表、小队单位、人物志、地形系统 |
| `游戏/程序代码/docs/` | 开发文档，定义数据结构、程序实现、AI、测试、日志 | 数据结构字段表、AI行为评分表、AI对战测试 |

## 本文档不定义玩法

本文档只描述程序如何存储和组织规则。

如果本文档与 `游戏/游戏设定/规则总表.md` 冲突，以 `游戏/游戏设定/规则总表.md` 为规则来源。

# 2. 表名前缀规范

## 2.1 设定表前缀：`cfg_`

`cfg_` 表示配置表、设定表、静态规则表。

特点：

- 由设定文档生成。
- 一般由开发者或策划维护。
- 对局过程中不频繁写入。
- 多数可被前端缓存。
- 可用于多局对战复用。

示例：

```text
cfg_games
cfg_squads
cfg_units
cfg_skills
cfg_statuses
cfg_terrains
```

## 2.2 对局记录表前缀：`match_`

`match_` 表示一次具体对局中的运行状态与事件记录。

特点：

- 每开一局生成。
- 对局过程中频繁读写。
- 存储单位当前位置、HP、AP、状态、据点归属、事件日志等。
- 可用于回放、AI训练、平衡测试。

示例：

```text
match_games
match_units
match_objectives
match_statuses
match_events
```

## 2.3 玩家与账号前缀：`user_`

`user_` 表示玩家账号、收藏、进度、设置。

特点：

- 与具体游戏规则分离。
- 不直接参与规则结算。
- 可跨多个游戏使用。

示例：

```text
user_profiles
user_progress
user_settings
```

## 2.4 多游戏资源前缀：`asset_`

`asset_` 表示跨游戏复用的美术、音效、动画、图标、背景等资源。

特点：

- 不直接属于《夕妖》规则。
- 可被未来多个游戏复用。
- 通过资源 key 被设定表引用。

示例：

```text
asset_files
asset_tags
asset_usages
```

## 2.5 AI与测试前缀：`sim_`

`sim_` 表示 AI 模拟、自动对战、平衡测试相关数据。

特点：

- 不影响正式玩家对局。
- 用于批量测试、胜率分析、机制平衡。
- 可读取 `cfg_` 设定表并生成 `match_` 对局记录。

示例：

```text
sim_runs
sim_agents
sim_match_results
sim_balance_metrics
```

| cfg_squads | 设定表 | 定义小队，例如青丘使团、天门执法队 | 是 |
| cfg_units | 设定表 | 定义英雄单位基础数值 | 是 |
| cfg_skills | 设定表 | 定义每名英雄的核心技能 | 是 |
| cfg_statuses | 设定表 | 定义状态，例如狐印、迷踪、逃逸、狐火充盈 | 是 |
| cfg_resources | 设定表 | 定义资源，例如 AP、气运、狐火 | 是 |
| cfg_dice_faces | 设定表 | 定义行动骰骰面 | 是 |
| cfg_terrains | 设定表 | 定义基础地形 | 是 |
| cfg_overlays | 设定表 | 定义覆盖层，例如狐火残留 | 是 |
| cfg_zones | 设定表 | 定义区域，例如勘验区 | 是 |
| cfg_objectives | 设定表 | 定义据点类型，例如中央据点、边缘据点 | 是 |
| cfg_decrees | 设定表 | 定义律令，例如禁行令、戒严令、追捕令 | 是 |
| cfg_maps | 设定表 | 定义自由对战与AI对战地图模板 | 是 |
| cfg_map_tiles | 设定表 | 定义地图模板中的格子、坐标、地形、据点 | 是 |

| match_squads | 对局表 | 记录本局双方选择的小队 | 是 |
| match_units | 对局表 | 记录本局单位实例，例如当前位置、HP、AP、是否已激活 | 是 |
| match_summons | 对局表 | 记录假身等召唤物实例 | 是 |
| match_team_resources | 对局表 | 记录队伍资源，例如青丘狐火、气运 | 是 |
| match_statuses | 对局表 | 记录单位身上的状态实例 | 是 |
| match_tiles | 对局表 | 记录本局地图格子实例 | 是 |
| match_overlays | 对局表 | 记录本局覆盖层实例，例如狐火残留 | 是 |
| match_zones | 对局表 | 记录本局区域实例，例如勘验区 | 是 |
| match_objectives | 对局表 | 记录本局据点归属 | 是 |

## 3.3 玩家与账号表：`user_`

| 表名 | 类型 | 用途 | 是否进入MVP |
|---|---|---|---|
| user_profiles | 用户表 | 记录用户基础资料 | 可选 |
| user_settings | 用户表 | 记录用户偏好设置 | 可选 |

MVP 若先做本地试玩或纯 AI 模拟，可以暂时不实现 `user_` 表。

## 3.4 多游戏资源表：`asset_`

| 表名 | 类型 | 用途 | 是否进入MVP |
|---|---|---|---|
| asset_files | 资源表 | 记录图片、音效、动画、图标等文件 | 可选 |
| asset_tags | 资源表 | 记录资源标签 | 可选 |
| asset_usages | 资源表 | 记录资源被哪些游戏、单位、技能引用 | 可选 |

`asset_` 表用于未来多游戏共用资源库，不与《夕妖》规则表混在一起。

## 3.5 AI模拟与平衡测试表：`sim_`

| 表名 | 类型 | 用途 | 是否进入MVP |
|---|---|---|---|
| sim_agents | 模拟表 | 定义AI类型，例如随机AI、规则评分AI、强化评分AI | 是 |
| sim_runs | 模拟表 | 记录一次批量模拟任务 | 是 |
| sim_match_results | 模拟表 | 记录每局AI对战结果 | 是 |
| sim_balance_metrics | 模拟表 | 汇总胜率、平均气运差、狐火触发次数等指标 | 是 |

`sim_` 表可以在实现 AI 对战测试时使用。

它不影响正式对局逻辑。

# 4. 表关系总览

## 4.1 游戏与规则版本

```text
cfg_games
1 对多
cfg_rulesets
```

一个游戏可以有多个规则版本。

---

## 4.2 规则版本与静态设定

```text
cfg_rulesets
1 对多
cfg_squads
cfg_units
cfg_skills
cfg_statuses
cfg_resources
cfg_dice_faces
cfg_terrains
cfg_overlays
cfg_zones
cfg_objectives
cfg_decrees
cfg_maps
cfg_map_tiles
```

每个规则版本拥有自己的一套静态设定。

后续如果规则改版，不直接覆盖旧数据，而是新增规则版本。

---

## 4.3 小队与单位

```text
cfg_squads
1 对多
cfg_units
```

一个小队包含多个英雄单位。

---

## 4.4 单位与技能

```text
cfg_units
1 对多
cfg_skills
```

当前 MVP 中，每个英雄只保留一个核心技能。

但关系设计为 1 对多，是为了未来支持普通技能、队长技能、被动技能和扩展技能。

当前实现可用约束限制：

```text
MVP 每个 unit 只启用 1 个 core skill
```

---

## 4.5 地图与格子

```text
cfg_maps
1 对多
cfg_map_tiles
```

一个地图模板包含多个六角格。

每个格子记录坐标、地形、是否为据点中心、是否为部署区。

---

## 4.6 地图与对局

```text
cfg_maps
1 对多
match_games
```

当前 MVP 中，地图直接服务自由对战、玩家 vs AI 和 AI vs AI。

不再经过 Campaign 或 Scenario。

---

## 4.7 对局与玩家方

```text
match_games
1 对多
match_players
```

一局游戏可以有多个参与方。

当前 MVP 通常是：玩家 vs AI、AI vs AI、玩家 vs 玩家。

---

## 4.8 对局与小队实例

```text
match_games
1 对多
match_squads

cfg_squads
1 对多
match_squads
```

`cfg_squads` 是静态小队定义。

`match_squads` 是本局内的小队实例。

---

## 4.9 对局小队与单位实例

```text
match_squads
1 对多
match_units

cfg_units
1 对多
match_units
```

`cfg_units` 定义单位基础数值。

`match_units` 记录本局单位当前 HP、位置、AP、是否已激活等。

---

## 4.10 单位实例与状态实例

```text
match_units
1 对多
match_statuses

cfg_statuses
1 对多
match_statuses
```

重复施加狐印时，不新增叠层，而是刷新持续时间。

---

## 4.11 对局与地图实例

```text
match_games
1 对多
match_tiles

cfg_map_tiles
1 对多
match_tiles
```

`cfg_map_tiles` 是地图模板。

`match_tiles` 是本局地图格子实例。

---

## 4.12 对局与据点实例

```text
match_games
1 对多
match_objectives

cfg_objectives
1 对多
match_objectives
```

`cfg_objectives` 定义中央据点、边缘据点类型。

`match_objectives` 记录本局据点归属。

据点归属不会因为单位离开而改变。

只有执行明确占领动作成功后才改变。

---

## 4.13 对局与覆盖层、区域

```text
match_games
1 对多
match_overlays

match_games
1 对多
match_zones
```

狐火残留进入 `match_overlays`。

勘验区进入 `match_zones`。

二者分开，是为了避免覆盖层和区域效果混淆。

---

## 4.14 对局与律令

```text
match_games
1 对多
match_decrees

cfg_decrees
1 对多
match_decrees
```

`match_decrees` 记录玄照本局何时颁布了哪条律令。

律令只影响颁布之后发生的事件。

因此必须记录：

```text
proclaimed_at_turn
proclaimed_at_action_index
```

---

## 4.15 对局与动作、事件、骰子

```text
match_games
1 对多
match_actions

match_actions
1 对多
match_events

match_actions
1 对多
match_dice_rolls
```

`match_actions` 记录玩家或 AI 选择了什么。

`match_events` 记录规则引擎实际产生了什么。

骰子结果单独进入 `match_dice_rolls`，方便回放和调试。

# 5. 当前推荐实现顺序

## 第一阶段：设定表

先实现：

```text
cfg_games
cfg_rulesets
cfg_squads
cfg_units
cfg_skills
cfg_statuses
cfg_resources
cfg_dice_faces
cfg_terrains
cfg_objectives
cfg_decrees
cfg_maps
cfg_map_tiles
```

目标：让规则能被数据库表达。

## 第二阶段：对局核心表

再实现：

```text
match_games
match_players
match_squads
match_units
match_team_resources
match_statuses
match_tiles
match_objectives
match_decrees
```

目标：让一局自由对战能跑起来。

## 第三阶段：回放与调试表

再实现：

```text
match_actions
match_events
match_dice_rolls
```

目标：让对局可回放、可调试、可做平衡分析。

## 第四阶段：AI模拟表

最后实现：

```text
sim_agents
sim_runs
sim_match_results
sim_balance_metrics
```

目标：让 AI 自动对战与平衡测试能批量执行。

# 6. 当前不建议混入的数据

以下内容不要放进当前 MVP 规则表：

- 单人战役
- 教学关卡
- 剧情流程
- 解锁奖励
- 宝物
- 百宝锦囊
- 妙计
- 阵型
- 法术
- 命运系统
- 天门威仪资源
- 皮肤
- 抽卡
- 经济系统
- 商城
- 用户充值

这些内容如需保留，应放入未来扩展或独立系统表。

# 7. 审计结论

当前建议采用五类前缀：

```text
cfg_    设定表
match_  对局记录表
user_   用户账号与设置表
asset_  多游戏资源表
sim_    AI模拟与平衡测试表
```

其中当前 MVP 最重要的是：

```text
cfg_
match_
sim_
```

`user_` 与 `asset_` 可以延后实现。

当前 `cfg_` 设定表不包含战役与关卡。

地图直接服务自由对战、玩家 vs AI 和 AI vs AI。

该结构能明确区分：

- 静态设定
- 对局运行状态
- 用户资料
- 多游戏资源
- AI模拟数据

避免把规则、资源、日志、用户数据混在同一批表里。
